# 反思：第三次重建为何仍然失败

## 一、我的三个错误

### 错误 1：没有真正读实验数据就动手

实验报告 81KB，我只搜索了关键词就下结论。我没有逐行阅读实验时间线，没有理解以下关键事实：

**实验验证的最小成功路径**：
1. 复合片段 → 缓存文件被剪映锁定（`private_importable` + locked）
2. 关闭主窗口（保留托盘）→ 文件解锁，root subdraft 出现
3. 重新启动剪映 → 重新打开项目 → 拖入成功

**关键状态变化**（重启前 vs 重启后）：
| 指标 | 重启前 | 重启后（打开项目） |
|------|--------|-----------------|
| 文件锁 | 🔒 被占用 | 🔒 重新被占用（新进程） |
| 缓存文件 UUID | EE96... | **616B...（新的！）** |
| root subdraft | 不存在 | **出现** |
| 可拖入 | ❌ | ✅ |

**我的错误**：我在 `_auto_detect_phase` 中用文件锁来判断状态，但重启后文件**又被新进程锁定**。所以重启后 GUI 扫描结果还是显示"需要重启"——完全错误。

真正的区别不是文件锁，而是：
- 重启后剪映建立了新的进程上下文
- root subdraft 已创建
- 同一个缓存文件在新进程上下文中可以被导入

**文件锁在重启后是不可靠的判断依据**。正确的方式是追踪**是否执行过重启操作**，而不是猜测文件状态。

### 错误 2：GUI 没有从用户角度设计

用户的使用流程应该是：

```
打开 GUI → 自动检测项目 → 自动扫描缓存 → 自动填好草稿名 → 用户只管点按钮
```

但我做的是：

```
打开 GUI → 手动选项目 → 手动点扫描 → 手动填草稿名 → 手动点操作
```

具体问题：

| 问题 | 用户期望 | 我的实现 |
|------|---------|---------|
| 项目选择 | 自动选最新的，需要时再切换 | 要手动点下拉框 |
| 重新检测后 | 自动选中最新项目 | 列表刷新但不选中 |
| 扫描后草稿名 | 自动从文件名生成 | 空白，要手动填 |
| 缓存状态显示 | 显示真实的可用性 | 第一次改成了全部"可用"，第二次改成看文件锁（也不对） |
| 重启后状态 | 自动知道已经重启了 | 靠文件锁猜（猜错） |

### 错误 3：没有尊重已验证的工作流

备份版有完整的工作流：
- `private_refresh.py`：重启后轮询等待缓存变为可导入（最多 45 秒）
- `private_prepare.py`：关闭主窗口 → 等待文件解锁 → 重新启动 → 返回 ready_for_manual_drag
- `source_resolver.py`：自动扫描 cloud_cache 镜像

这些功能在当前版本被简化掉了，但没有替代方案。

## 二、根因分析

**根本原因：我试图在不理解领域知识的情况下修改代码。**

具体表现：
1. 没有读完实验报告就动手 → 对状态转换的理解有误
2. 用通用编程思维（文件锁检测）替代领域知识（进程上下文切换） → 判断逻辑错误
3. 没有把自己当用户来测试 GUI → 交互设计不合理
4. 每次修一个问题就引入新问题 → 补丁式开发

## 三、正确的做法

### 3.1 状态追踪：用操作历史，不猜文件状态

```python
# 正确的方式：记录用户做了什么
workflow_phase: WorkflowPhase  # 由操作驱动，不靠文件猜测

# 重启按钮点击 → phase = RESTARTED（这是事实，不需要验证）
# 不要在扫描时用文件锁去"猜"用户是否重启过
```

### 3.2 GUI 设计原则：最小用户操作

1. **启动时**：自动检测项目 → 自动选最新 → 自动扫描 → 自动填草稿名
2. **每次状态变化后**：自动更新所有相关 UI（不需要用户手动刷新）
3. **下拉框刷新后**：自动选中第一项（最新项目）

### 3.3 缓存分类：只展示格式分类，不判断运行时可用性

```
✓ 标准 MP4          — MediaInfo 可解析
✓ 私有缓存可导入     — 有 alpha sidecar
✗ 私有缓存不可导入   — 缺 alpha 或格式异常
```

"需不需要重启"由工作流阶段决定（用户是否已点过重启按钮），不由文件锁决定。
